Software Architecture Summary I

Here is the note of Software Architecture book written by Medvidovic, including some basic concepts, basic terminologies, common software architecture, good examples, software architecture analysis approach and so on.

#Basic Concepts

###What is Software Architecture?
[1]Definition
A software system’s architecture is the set of principal design decisions about the system and the blueprint for a software system’s construction and evolution.

[2]Design decisions encompass every facet of the system under development:

  • Structure
  • Behavior
  • Interaction
  • Non-functional properties

[3]“Principal” implies a degree of importance that grants a design decision “architectural status”, which means that not all design decisions are architectural.

[4]Other Definitions of Software Architecture

  • Perry and Wolf
    Software Architecture = { Elements, Form, Rationale } (What, How, Why)
  • Shaw and Garlan
    Software architecture [is a level of design that] involves the description of elements from which systems are built, interactions among those elements, patterns that guide their composition, and constraints on these patterns.
  • Kruchten
    Software architecture deals with the design and implementation of the high-level structure of software, dealing with abstraction, decomposition, composition, style, and aesthetics.

[5]Temporal Aspect

  • Design decisions are and unmade over a system’s lifetime, meaning that architecture has a temporal aspect.
  • At any given point in time the system has only one architecture.
  • A system’s architecture will change over time.

###Prescriptive Architecture
A system’s prescriptive architecture captures the design decisions made prior to the system’s construction, which is the as-conceived or as-intended architecture.

###Descriptive Architecture
A system’s descriptive architecture describes how the system has been built, which is the as-implemented or as-realized architecture.

###Architectural Evolution
[1]When a system evolves, ideally its prescriptive architecture is modified first; but in practice, the system – and thus its descriptive architecture – is often directly modified.
[2]This happens because of

  • Developer sloppiness
  • Perception of short deadlines which prevent thinking through and documenting
  • Lack of documented prescriptive architecture
  • Need or desire for code optimizations
  • Inadequate techniques or tool support

###Architectural Degradation
[1]Architectural drift
Introduction of principal design decisions into a system’s descriptive architecture that are not included in, encompassed by, or implied by the prescriptive architecture but which do not violate any of the prescriptive architecture’s design decisions.

[2]Architectural erosion
Introduction of architectural design decisions into a system’s descriptive architecture that violate its prescriptive architecture.

###Architecture Recovery
The process of determining a software system’s architecture from its implementation-level artifacts.

###Deployment
[1]Definition
Executable modules are physically placed on the hardware devices on which they are supposed to run.

[2]The deployment view of an architecture can be critical in assessing whether the system will be able to satisfy its requirements. And possible assessment dimensions include:

  • Available memory
  • Power consumption
  • Required network bandwidth

###Software Architecture’s Elements
[1]A software system’s architecture should be a composition and interplay of different elements, which may include:

  • Processing
  • Data, also referred as information or state
  • Interaction

####Components
[1]Definition
A software component is an architectural entity that:

  • encapsulates a subset of the system’s functionality and/or data;
  • restricts access to that subset via an explicitly defined interface;
  • has explicitly defined dependencies on its required execution context.

[2]Components typically provide application-specific services

####Connectors
[1]Definition
A software connector is an architectural building block tasked with effecting and regulating interactions among components.

[2]Connectors typically provide application-independent interaction facilities

[3]Examples of Connectors

  • Procedure call connectors
  • Shared memory connectors
  • Message passing connectors
  • Streaming connectors
  • Distribution connectors
  • Wrapper/adaptor connectors

[4]Implemented vs. Conceptual Connectors
(1)Connectors in software system implementations

  • Frequently no dedicated code
  • Frequently no identity
  • Typically do not correspond to compilation units
  • Distributed implementation
    (2)Connectors in software architectures
  • First-class entities
  • Have identity
  • Describe all system interaction
  • Entitled to their own specifications & abstractions

[5]Roles
(1)Locus of interaction among set of components

(2)Protocol specification (sometimes implicit) that defines its properties

(3)Roles

  • Communication
  • Coordination
  • Conversion
  • Facilitation

[6]Connectors as Communicators
(1)Main role associated with connectors

(2)Supports

  • Different communication mechanisms
  • Constraints on communication structure/direction
  • Constraints on quality of service

(3)Separates communication from computation

(4)May influence non-functional system characteristics

[7]Connectors as Coordinators
(1)Determine computation control

(2)Control delivery of data

(3)Separates control from computation

(4)Orthogonal to communication, conversion, and facilitation

[8]Connectors as Converters
(1)Enable interaction of independently developed, mismatched components

(2)Mismatches based on interaction

(3)Examples: Adaptors and Wrappers

[9]Connectors as Facilitators
(1)Enable interaction of components intended to interoperate

(2)Govern access to shared information

(3)Ensure proper performance profiles

(4)Provide synchronization mechanisms

[10]Connector Types

  • Procedure call (multi connectors added together)
  • Data access
  • Event
  • Stream
  • Linkage
  • Distributor
  • Arbitrator
  • Adaptor

[11]How to select a connector

  • Determine a system’s interconnection and interaction needs
    Software interconnection models can help
  • Determine roles to be fulfilled by the system’s connectors
    Communication, coordination, conversion, facilitation
  • For each connector
    Determine its appropriate type(s)
    Determine its dimensions of interest
    Select appropriate values for each dimension
  • For multi-type, i.e., composite connectors
    Determine the atomic connector compatibilities

[12]Software Interconnection Models

  • Unit interconnection
  • Syntactic interconnection
  • Semantic interconnection

[13]Unit Interconnection
(1)Defines relations between system’s units

  • Units are components (modules or files)
  • Basic unit relationship is dependency

(2)Characteristic

  • Coarse-grain interconnections
  • Interconnections are static
  • Does not describe component interactions

[14]Syntactic Interconnection
(1)Describes relations among syntactic elements of programming languages

  • Variable definition/use
  • Method definition/invocation

(2)Characteristic

  • Finer-grain interconnections
  • Interconnections are static & dynamic
  • Incomplete interconnection specification

[14]Semantic Interconnection
(1)Expresses how system components are meant to be used; Captures how system components are actually used; Interconnection semantics can be formally specified.

(2)Characteristics

  • Builds on syntactic interconnections
  • Interconnections are static & dynamic
  • Complete interconnection specification
  • Necessary at level of architectures
  • Ensuring other properties of interaction

[15]Composite Connectors
(1)The composition can be considered at the level of connector type dimensions and subdimensions.

(2)Examples:

  • Grid connectors
  • Peer-to-peer connectors
  • Client-server connectors
  • Event-based connectors

###Adaptor
[1]Adaptation
Modification of a software system to satisfy new requirements and changing circumstances.

[2]Motivation

  • Corrective Changes
  • Modification to the functional requirements
  • New or changed non-functional system properties
  • Changed operating environment
  • Observation and analysis

[3]Architecture-Centric Adaptation
(1)In the absence of explicit architecture the engineer is left to reason about adaptation from memory or source code.

(2)Architecture can serve as the primary focus of reasoning about potential adaptations

[3]Activities, Agents, and Entities
(1)Adaptation management and evolution management can be separated into three types of activities.

  • Strategic refers to determining what to do
  • Tactical refers to developing detailed plans for achieving the strategic goals
  • Operations refers to the nuts-and-bolts of carrying out the detailed plans

[4]Categories of Techniques

  • Observing and collecting state
  • Analyzing data and planning change
  • Describing change descriptions
  • Deploying change descriptions
  • Effecting the change

[5]Architectures/Styles Supporting Adaptation

  • Application programming interfaces (API)
  • Scripting languages
  • Plug-ins
  • Component/object architectures
  • Event interfaces

[6]The Special Problems of On-the-fly Change
(1)The principle of quiescence

  • A node in the active state can initiate and respond to transactions.
  • The state identified as necessary for reconfiguration is the passive state: node must respond to transaction; not currently engaged in a transaction that it initiated; it will not initiate new transactions.

####Configurations
[1]Definition
An architectural configuration, or topology, is a set of specific associations between the components and connectors of a software system’s architecture.

###Architectural Styles
[1]Definition
An architectural style is a named collection of architectural design decisions that:

  • are applicable in a given development context;
  • constrain architectural design decisions that are specific to a particular system within that context;
  • elicit beneficial qualities in each resulting system.

[2]Benefits of using styles

  • Design reuse: Well-understood solutions applied to new problems.
  • Code reuse: Shared implementations of invariant aspects of a style.
  • Understandability of system organization: A phrase such as “client-server” conveys a lot of information.
  • Interoperability: Supported by style standardization.
  • Style-specific analyses: Enabled by the constrained design space.
  • Visualizations: Style-specific depictions matching engineers’ mental models.

###Architectural Patterns
[1]Definition
An architectural pattern is a set of architectural design decisions that are applicable to a recurring design problem and parameterized to account for different software development contexts in which that problem appears.

[2]Style vs. Pattern

  • A style prescribes the types of system elements and constraints on their composition.
  • A pattern prescribes a specific set of system elements and their specific organization in (part) of a system.

###Architectural Models, Views, and Visualizations

####Architecture Model
[1]Definition
An artifact documenting some or all of the architectural design decisions about a system.

####Architecture Visualization
[1]Definition
A way of depicting some or all of the architectural design decisions about a system to a stakeholder.

####Architecture View
[1]Definition
A subset of related architectural design decisions.

###Stakeholders in a System’s Architecture

  • Architects
  • Developers
  • Testers
  • Managers
  • Customers
  • Users
  • Vendors

###Architecture Design
[1]Abstraction
The act or process of separating in thought, of considering a thing independently of its associations; or a substance independently of its attributes; or an attribute or quality independently of the substance to which it belongs.

[2]Reification
The mental conversion of … [an] abstract concept into a thing.

[3]Deduction
The process of drawing a conclusion from a principle already known or assumed; spec. in Logic, inference by reasoning from generals to particulars; opposed to INDUCTION.

[4]Induction
The process of inferring a general law or principle from the observation of particular instances (opposed to DEDUCTION, q.v.).

[5]Simple Machines
Simple machines serve as an abstraction of a potential system that will perform the required task and provide a plausible first conception of how an application might be built.
Every application domain has its common simple machines.

[6]Separation of Concerns
Separation of concerns is the subdivision of a problem into (hopefully) independent parts.

###Architecture Visualization
An architectural visualization defines how architectural models are depicted, and how stakeholders interact with those depictions, whereas depition is a picture or other visual representation of design decisions and interaction is mechanism allows stakeholders to interact with design decisions in terms of the depiction.

###Architecture Implementation Frameworks
An architecture implementation framework is a piece of software that acts as a bridge between a particular architectural style and a set of implementation technologies. It provides key elements of the architectural style in code, in a way that assists developers in implementing systems that conform to the prescriptions and constraints of the style.

###Domain-Specific Software Architectures
[1]Definition
A DSSA is an assemblage of software components that:

  • specialized for a particular type of task (domain),
  • generalized for effective use across that domain, and
  • composed in a standardized structure (topology) effective for building successful applications.

[2]Since DSSAs are specialized for a particular domain they are only of value if one exists for the domain wherein the engineer is tasked with building a new application and the pre-eminent means for maximal reuse of knowledge and prior development and hence for developing a new architectural design.

[3]Three Key Factors
(1)Domain
Must have a domain to constrain the problem space and focus development.

(2)Technology
Must have a variety of technological solutions—tools, patterns, architectures & styles, legacy systems—to bring to bear on a domain

(3)Business
Business goals motivate the use of DSSE

  • Minimizing costs: reuse assets when possible
  • Maximize market: develop many related applications for different kinds of end users

(4)Domain + Business
Corporate Core Competencies: Domain expertise augmented by business acumen and knowledge of the market.

(5)Domain + Technology
Application Family Architectures:

  • All possible technological solutions to problems in a domain.
  • Uninformed and unconstrained by business goals and knowledge.

(6)Business + Technology
Domain independent infrastructure: Tools and techniques for constructing systems independent of any particular domain.

(7)Domain + Business + Technology
Domain-specific software engineering: Applies technology to domain-specific goals, tempered by business and market knowledge.

###Domain Model
A domain model is a set of artifacts that capture information about a domain.

  • Functions performed
  • Objects (also known as entities) that perform the functions, and on which the functions are performed
  • Data and information that flows through the system

###Info Model
[1]Context Info Diagram

  • Defines high-level entities
  • Defines what is considered inside and outside the domain (or subdomains)
  • Defines relationships and high-level flows

[2]Entity-Relationship Diagram
Defines entities and cardinal relationships between them

[3]Object Diagram

  • Defines attributes and operations on entities
  • Closely resembles class diagram in UML but may be more abstract

###Feature Model
[1]Feature Relationship Diagram

  • Describes overall mission operations of a system
  • Describes major features and decomposition

[2]Use Case Diagram

  • Defines use cases within the domain
  • Similar to use case models in UML

[3]Representation Diagram
Defines how information is presented to human users

###Feature Model
[1]Representation Diagram
Defines how information is presented to human users

###Operational Model
[1]Data Flow Diagram
Focuses on data flow between entities with no notion of control

[2]Control Flow Diagram
Focuses on control flow between entities separate from data flow

[3]State Transition Diagram

  • Focuses on states of systems and transitions between them
  • Resembles UML state diagrams

###Reference Architecture
[1]Definition
Reference architecture is the set of principal design decisions that are simultaneously applicable to multiple related systems, typically within an application domain, with explicitly defined points of variation.

[2]Examples:

  • Structural view of Lunar Lander DSSA.
  • Invariant with explicit points of variation.

###Product Lines
[1]A set of related products that have substantial commonality. One potential ‘silver bullet’ of software engineering,

###Product Line Architecture
[1]Definition:
A product-line architecture captures the architectures of many related products simultaneously

[2]Product Component Table
Helps us decide whether creating a product line is viable or feasible

[3]Group Components into Features

  • Not a mechanical process
  • Attempt to identify (mostly) orthogonal features, or features that would be beneficial in different products
  • Not a mechanical process
  • Attempt to identify (mostly) orthogonal features, or features that would be beneficial in different products

[4]Reconstitute Products from Features
Use technical and business knowledge to identify which combinations form feasible or marketable products that will be constructed

###Non-Functional Property
[1]Definition
A software system’s non-functional property (NFP) is a constraint on the manner in which the system implements and delivers its functionality.

[2]Examples

  • Efficiency
  • Complexity
  • Scalability
  • Heterogeneity
  • Adaptability
  • Dependability

###Deployment
[1]Definition
Deployment is the process of placement of a system’s software components on its hardware hosts.

###Mobility
[1]Definition
Changing the deployment of a component during runtime is called migration or redeployment;
Migration or redeployment is a type of software system mobility.

[2]Categories
(1)Movement of software across hardware hosts during the system’s execution, that action is referred to as code mobility or logical mobility
(2)If a software module that needs to be migrated contains runtime state, then the module’s migration is known as stateful mobility
(3)If only the code needs to be migrated, that is known as stateless mobility

Playing with numbers

Sort the given numbers in non-descending order; then for every query number, adding the current number to sum of all previous query number; after that, if the sum is negative, using binary search to find the lower bound of the absolute value of the sum only in the positive part of the array; if the sum is positiive, using binary search to find the upper bound of the negative value of the sum only in the negative part of the arrary. At last, calculate the absolute value of the sum of the lower/upper bound part adding the sum of query number and the sum of other parts adding the sum of query number and finally, calculate the absolute value of the sum of all these two parts.

System Design Interview Summary

Here is the note of this tutorial: HiredinTech-System Design. It includes the guidlines about how to think in system design interview problem, real world system design examples, system deisgn special concepts and additional notes.

At first, here is the video of system design course from Harvard University, which teaches some basic and interesting ideaof how to do the system design.

Then, comes the note.

###Introduction
[1]Two major challenges most candidates face:

  • Approach questions in a chaotic way and get ratholed.
  • Lack solid understanding of how to properly design architectures that scale.

[2]Scalability Fundamentals (Concepts)

  • Vertical scaling
  • Horizontal scaling
  • Caching
  • Load balancing
  • Database replication
  • Database partitioning
  • Database sharding

[3]Two Major Bottlenecks of System

  • Handling a lot of users
  • Handling a lot of data

[4]Concepts should be mastered

  • HAProxy (software load balancing)
  • nginx (web server, different from tomcat)
  • memcached (a tool for handling high write data like view counters)
  • Redis (NO SQL Server)
  • Lucene
  • RabbitMQ (a tool for job system)
  • Munin
  • NodeJS

###System Design Process
[1]Clarify the constraints and use cases
The very first thing is to clarify the system constraints and to identify what use cases the system needs to satisfy. Spend a few minutes questioning your interviewer and agreeing on the scope of the system.
You will also have to think about the use cases that are expected to occur.
Use cases -> 需求
Constraints -> 系统要达到的上限（功能要求达到的上限）
For constraints, you should do some quick math, or some statistics, in the brain to scope the system.

[2]Do abstract design
Outlining a high-level abstract design. The goal of this step is to outline all the important components that the architecture will need.

You can tell the interviewer that you would like to do that and draw a simple diagram of your ideas. Sketch your main components and the connections between them. Don’t get lured to dive deep into some particular aspect of the abstract design. Rather, make sure you sketch the important components and the connections between them.

Usually, this sort of high-level design is a combination of well-known techniques, which people have developed. You have to make sure you are familiar with what is out there and feel comfortable using this knowledge.

[3]Understand bottlenecks
Start thinking about what bottlenecks the high-level design has. Perhaps your system needs a load balancer and many machines behind it to handle the user requests. Or maybe the data is so huge that you need to distribute your database on multiple machines.

These are just examples of questions that you may have to answer in order to make your solution complete. It may be the case that the interviewer wants to direct the discussion in one particular direction. Then, maybe you won’t need to address all the bottlenecks but rather talk in more depth about one particular area. In any case, you need to be able to identify the weak spots in a system and be able to resolve them.

Remember, usually each solution is a trade-off of some kind. However, the important thing is to be able to talk about these trade-offs, and to measure their impact on the system given the constraints and use cases defined.

Once you have outlined the core bottlenecks you see, you can start addressing them in the next step.

[4]Scaling your abstract design (让你的设计更加具体化)
Once you are ready with your high-level design and have made sure that the interviewer is ok with it, you can dive into making it more detailed.

Usually, this means making your system scale.

[5]Summary
To approach any system design question.
First build a high-level architecture by identifying the constraints and use cases, sketching up the major components and the relationships between them, and thinking about the system bottlenecks.
Then apply the right scalability patterns that will take care of these bottlenecks in the context of the system constraints.
Having learned a repeatable approach for system design, as well as seen lots of real-life examples of scalable architectures, your chances of acing the tech interview are very solid.

###Practice System Design Problems
[1]Come up with problems and discuss their solutions with friends
Look at the massive systems you use every day developed by the companies you are interviewing for. What systems did these companies have to design? Create a list of the fundamental system design questions that bubble up in your brain.
Start by compiling a list of 10 to 20 of these questions. Then, pick one and try to come up with an efficient design.
Once you’re ready, find a knowledgeable friend or coworker, go to a meeting room and start discussing ideas. It’d be tons of fun and you’ll learn a lot from the process.
Don’t spend hours: initially, limit your time per question to one hour and ultimately try to fit every question in 20 to 30 minutes.

[2]Do Mock Interview
Mock interviews are a fantastic way to practice system design questions. System design questions are all about organic discussions around trade-offs. Try to find an unbiased and knowledgeable person, and do several interviews with them. It can be a friend, a colleague, or you could use a paid service.

Recommendation is to start by working on system design questions yourself, then move on to discussing designs with friends, to finally doing mock interviews.

###Real World Examples
*Instagram
Note:
Keep it very simple.
Do not re-invent the wheel.
Go with proven and solid technologies when you can.

*cinchcash
Note:
“Be brief, be bright, be gone” : Respect another person’s time.
Don’t come with problems, come with solutions.
Don’t go chasing hot technologies of the day. Instead ‘mitigate your top problems’. We adopt new technologies but do so, when the business case requires it. Appetite for Production outages decreases significantly when you have millions of users.
Achieve “essential”, then worry about “excellent”.
Be a “how team” instead of a “no team”.
Build security into the software development lifecycle.

*Justin.Tv

*Playfish
Note:
Using SOA (Service Oriented Architecture).
Using sharded architecture for the write heavy service.
Using BLOB (Binary Large Object) in MySQL to store the user data.
Asynchronicity. Writing on the server is asynchronous from game play.
Agility. A common thread through all of Playfish thinking is the relentless need for agility, to be able to respond quickly, easily, and efficiently to every situation.
Applying the principle of loose coupling as much as possible.
Agility + Data + Design = allows new inspired features to be tried out quickly to see if they work or not.
Build an architecture that takes advantage of the special nature of your application.

Note:
Bank of very simple, stateless web servers, running simple Java and Velocity templates.
Bank of “services” - each service is high level, business oriented API optimized for over the wire performance.
Web servers are arranged in “pools” to allow for A/B testing, and control over deployment.
Numerous test frameworks available: Unit, Functional, Load, Smoke, Selenium, Load, and a test lab.
Keep designs simple and focused on the near term business needs - do not design too far ahead.
You can go very far in scaling a database vertically, especially if you minimize use of joins, and especially if you can fit everything into RAM.
Avoid joins when possible.

Software - keep it really, really, really simple.
Process. Less is better. You need to use source code control.

###Ugly Number
Use min heap. Initially, we push 3, 5, and 7 into heap and set counter as 0. Before counter reaches the upper bound, every time we pop out tht top number and multiply it by 3, 5 and 7 respectively and then push the corresponding numbers back to min heap, and increase the counter by one. The last number poped out from heap is answer.
小顶堆题目。初始化时将3，5和7插入堆中，然后设置计数器为1。在计数器的值小于阀值时，每次取出一个数，计数器就加一，然后将该数字分别乘以3，5和7，得到的结果存入到堆中。跳出循环前的出堆的最后一个数，就是答案。
