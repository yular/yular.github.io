Algorithm Contest Online Judge List

For OIer and ACMer, to win award in the algorithm contest, they have to find Online Judge (OJ) to do the training before the contest. Training is usually hard and boring, but worthy. So here is the list of OJ that may be useful for those who want to pass the alogrithm interview, or would like to win in the algorithm contests. Hope it is useful.

For Interview:
1.LeetCode
Links: https://leetcode.com
Easy but most useful and widely used.

2.LintCode
Links: https://lintcode.com
Easy and harder than LeetCode.

3.Codility
Link: https://codility.com/
Offer programming training and contest. Harder than LintCode.

4.Hackerrank
Link: https://www.hackerrank.com/
Offer training and contest, too. Have more modules than codility, including AI technique, Functional programming, Shell programming and etc. In fact, the algorithm problems are much more difficult than average interview ones. ACMer and OIer can use them in the training as well.

For OIer and ACMer
4.Hackerrank
Link: https://www.hackerrank.com/
The same description as above.

5.Topcoder
Link: http://www.topcoder.com/
One of the most famous and challenging algorithm contest platform. Take the SRM and enjoy it! Go to the TCO OPEN every year and try to get the T-shirt and even the onsite contest! The platform has the most talented and skillful programmers in the world!

6.Codeforces
Link: http://codeforces.com/
Another famous and challenging algorithm contest platform. It is always going with TCO. Take the Codeforces Contest Round and I am sure that you will enjoy it soon!

  1. USACO
    Link: http://train.usaco.org/usacogate
    One of the most famous training OJ for OIers in the world. Its contests are also in high quality. But the link may expire or change. But do not worry. You can Google USACO and then will find it. The problems there are interesting and challenging.

8.UVA
Link: https://uva.onlinejudge.org/
One of the oldest and most challenging OJ in the world. A pretty good training OJ for ACMers. The problems are difficult and can help you learn a lot. Most importantly, it contains all the ACM Regional Contest problems and World Final problems every year. And it holds a lot of contests to help ACMers to prepare for their up coming regional or world final contest.

9.SGU
Link: http://acm.sgu.ru/
One of the most challenging OJ in the world. It is powered by a Russian university. But sometimes I think it is the most difficult OJ in the world … It only contains hundreds of problems and each of them is not easy and even rather difficult to solve. If you want to challenge yourself, take it :-)

10.Timus
Link: http://acm.timus.ru/
Another OJ powered by Russian university. The problem there is also difficult but easier than those in SGU. But they are also in high quality and the test data may be tricky and strict! Be cautious!

11.SPOJ
Link: http://www.spoj.com/
One of the oldest but challenging OJ in the world. It is difficult to pass the test data of problems. You should try to optimize your program as much as possible.

12.POJ
Link: http://poj.org/
One of the most famous OJ in China. There are many problems but some of them can be solved using almost the same programs … Very interesting , right? Anyway, the quality of the problems and the test data are still high.

13.HDU OJ
Link: http://acm.hdu.edu.cn/
I think it is the most famous OJ in China because it has the largest number of problems and every year, the online regional contests are held in it. Since there are too many problems, some of them of course are in low quality. But there are many problem recommendation list on the Internet. Just find one of them and solve the problems in the list. I think the same approach can be applied to any OJ as it is impossible for most of the people to solve all the problems in an OJ.

14.ZOJ
Link: http://acm.zju.edu.cn/onlinejudge/
One of the most famous OJ in China. The problems in this OJ are still pretty good. It sometimes holds ZOJ Monthly Contests. ACMers should take these contests I think because the problems there are challenging.

15.CodeChef
Link: http://www.codechef.com/
An contest OJ similar to Codeforces and TopCoder. The problems there are challenging.

Object-Oriented Design Pattern Summary

This summary is the note of Head First Design Pattern, including some basic knowledge, guidelines, and all the design patterns and their application examples described in the book.

Here is the link of Head First Design Pattern in Amazon:
Head First Design Pattern

###Basic

#####1. Basic Knowledge
[1]Design Pattern Concept
A Pattern is a solution to a problem in a context.

[2]Pattern Categorites

  • Creational patterns
  • Structural patterns
  • Behavioral patterns

[3]Object Oriented Basic

  • Abstraction
  • Encapsulation
  • Polymorphism
  • Inheritance.

[4]Object Oriented Principles

  • Encapsulate what varies; Favor composition over inheritance; Program to interfaces, not implementation.
  • Strive for loosely coupled designs betweens objects that interact. (The two objects have very little knowledge of each other)
  • Classes should be open for extension but closed for modification.
  • Dependency Inversion Principle: Depend upon abstraction. Do not depend on concrete classes.
  • Least Knowledge: Talk only to your immediate friends.
    Note: Only invoke methods that belongs to: the object itself; objects passed in as a parameter to the method; Any object the method creates or instantiates; Any components of the object.
  • Hollywood Principle: Do not call us, we will call you.
  • A class should have only one reason to change.

[5]Tips:

  • Design Patterns give us a shared vocabulary with other developers and let us think at the pattern level.
  • A design guru thinks about how to create flexible designs that are maintainable and that can cope with change.

###GuideLines

#####2. Guidelines to Follow the Dependency Inversion Principle

  • No variable should hold a reference to a concrete class.(Use a factory to get around that)
  • No class should derive from a concrete class
  • No method should override an implemented method of any of its base classes.

#####3. Pattern Design Writers Guidelines
[1]To be a Design Pattern Writer

  • Do your homework
  • Take time to reflect and evaluate
  • Get your ideas down on paper in a way others can understand
  • Have others try your patterns then refine and refine some more
  • Do not forget the rule of three

[2]Better Living with Design Pattern

  • Learn the all too common misconceptions about the definition of a “Design Pattern”
  • Discover those unify Design Pattern Catalogs and why you just have to get one
  • Avoid the embarrasment of using a Design Pattern at the wrong time
  • Learn how to keep patterns in classifications where they belong
  • See that discovering patterns is not just for the guius: read out quick How To and become a patterns writer too
  • Be there when the true identity of the mysterious Gang of Four is revealed
  • Keep up with the neighbours - the coffee table books any patterns user must down
  • Learn to train your mind a Zen master
  • Win friends and influence developers by improving your patterns vocabulary

[3]Top five ways to share your vocabulary

  • In design meetings
  • With other developers
  • In architecture documentation
  • In code comments and naming conventions
  • To groups of interested developers

[4]Thinking in Patterns

  • Keep it Simple
  • Design Patterns are not a magic bullet; In fact, they are not even a bullet
  • You know you need a pattern when you find that a simple solution does not work.
  • Refactoring time is Patterns time!
  • Take out what you do not really need. Do not be afraid to remove a Design Pattern from the design
  • If you do not need it now, do not do it now

[5]Mind on Patterns

  • Beginner Mind: The beginner uses patterns everywhere.
  • Intermediate Mind: As learning progresses, the intermediate mind starts to see where patterns are needed and where they are not.
  • Zen Mind: Be able to see where patterns fit naturally.

###Design Patterns

#####4. The Strategy Pattern
The Strategy Pattern defines a family of algorithms encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from clients that use it.

Example: Duck Behaviors Design

Tips:
[1]Identify the aspects of application that vary and separate them from what stays the same.

  • “Program to an interface” really means “Program to a supertype”. (Polymorphism)
    When abstract superclass or when interface?
  • Relationship between classes: HSA-A, IS-A and IMPLEMENTS
  • The promise of the object-oriented design(way) is reuse.

#####5. The Observer Pattern
Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

Note: Loosely coupled is needed.

#####6. The Decorator Pattern
Attach additional responsibility to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

Example: Java IO classes

Note: Nesting construction

#####7. The Factory Pattern
Defines an interface for creating an object, but less subclasses decide which class to instantiate. Factory Method lets a class defer instantiation to subclasses.

Note:

  • Creator Class and Product Class, Parallel Class Hierarchies;
  • Encapsulating object creation;
  • Depend on abstraction instead of concrete classes;
  • Subclass

#####8. The Abstract Factory Pattern
Provides an interface for creating families of related or dependent objects without specifying their concrete classes.

Note:

  • Often implemented as factory methods;
  • Used when have families of products needed to create

#####9. The Singleton Pattern
Ensures a class has only one instance and provides a global point of access to it.

#####10. The Command Pattern
Encapulation a request as an object, thereby letting you parameterize clients with different requests, queue or log requests and support undoable operation.

Note:
Support Queue data structure.

#####11. The Adapter Pattern
Converts the interface of a class into another interface the clients expect. Adapter lets classes work together that could not otherwise because of incompatible intefaces.

Example: Home Theater

#####12. The Facade Pattern
Provides a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.

Note:
To make a system easier to use through a simplied interface.

#####13. The Template Method Pattern
Defines the skeleton of an algorithm in a method, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the structure of algorithm.

Example: JFrames swinging

#####14. The Iterator Pattern
Provides a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

Example: Java.util Iterator, Java Collection Framework.

#####15. The Composite Pattern
Allows you to compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

#####16. The State Pattern
Allows an object to alter its behavior when its internal state changes. The object will appear to change its class.

#####17. The Proxy Pattern
Provides a surrogate or placeholder for another object to control access to it.

Example: Java.rmi; Remote Proxy && Virtual Proxy && Protect; Proxy-Java Reflection.

#####18. The Compound Pattern (Pattern of Pattern)
A Compound Pattern combines two or more patterns into a solution that solves a recurring or general problem.

Example: Model-View-Control.

#####19. The Anti-Pattern
How to go from a problem to a BAD solution

Extra: Some Simple Steps of Designing Gumball Machine
[1]Define a State interface that contains a method for every action in the Gumball Machine.
[2]Then we are going to implement a State class for every state of the machine. These classes will be responsible for the behavior of the
machine when it is in the corresponding state.
[3]Finally, we are going to get rid of all of our conditional code and instead delegate to the state class to do the work for us.

Interview Summary: Behavior Question

The following is the list of common Behavior Questions in the IT interview and this page will be updated at any time:

(1) Short self-introduction.(Should be less than one minute)

(2) Talk about your most challenging project

(3) Talk about your favourite/most interesting/most enjoying project

(4) Talk about the most difficult/challenging bugs/difficulty you meet in the project

(5) In your team, if your teammates have different ideas about the project development, assume you are the leader, how to resolve the disagreement?

(6) You can ask some questions (about (my) company/team/work …) now (usually during the last several minutes of the interview)

(7) Here is our xxxxx product. How do you think of it? Any good/bad features? (Usually we should talk more about the bugs or bad features of the product)

(9) Why you choose our company?

(10) Why you choose this job?

(11) What you have learned from your project?

(12) How much of your day do you spend coding?

(13) How many meetings do you have every week?

(14) What is the ratio of testers to developers to program managers? What is the interaction like? How does project planning in the team?

(15) I notice that you use technology X. How do you handle problem Y?

(16) Why did the product choose to use X protocol over Y protocol? I know it has the benefits like A, B, C, but many companies do not use it because of issue D.

(17) I am very interested in scalability. Did you come in with a background in this, or what opportunities are there to learn about it?

(18) I am not familiar with technology X, but it sounds like a very interesting solution. Could you tell me a bit more about how it works?

General Approach to Solve Algrithm Problems

This blog is the general approach to solve algorithm problems based on my own experience. Hope it will be helpful for those who participate algorithm contest like TopCoder, CodeForce, OI, ACM-ICPC and so on or have to take technical interviews involving algorithms.

  1. Read the Problem Statement
    Read the problem statement very very very carefully. Do remember that. Understand the following information:
    (1) What is the meaning of the problem statement.
    (2) What does the problem ask we to do. To count the cases? To find the maximum/minimum value?
    (3) What are the constraints of the problem? What is the range of the input? 1<= N <= 100 or 1 <= N <= 1000?
    (4) What is the input format and output format of the problem?
    (5) Are both the sample input and output correct?
    If the problem statement is difficult to understand, ask the problem writter or those who can offer a hand.

  2. Build the Mathmatical Model and Design one or more Algorithms
    After fully understand the problem, we should know how to calculate the sample output given sample input. This step is to find the correct mathematical models to solve the problem. When we find the correct mathmetical models, then we can start to design algorithms that can solve the models correctly (Note: not necessarily efficiently).

  3. Validate the Algorithm
    After the algorithm is proposed, we should then validate it. The tricks are to tet the algorithm with large test case, corner test case, tricky test case and general test case. Make sure that the algorithm are correct and efficient in all these kinds of cases so that we can avoid Wrong Answer, Time Limit Excceed, Memory Limit Excceed or other incorrect message returned by online judege.

  4. Optimize the Algorithm
    If our algorithm can pass all the above cases, congratulations! Our algorithm can receive Accept from online judege. But in some cases, it may not be efficient enough and online judege will return TLE for us. In that case, we should optimize the algorithm. There are mainly two cases:
    (1) We may have to rebuild the mathematical model so that more efficient algorithms can be used.
    (2) We only have to use other algorithms for the current model but more efficient techniques should be applied.
    Sometimes, we can optimize our algorithm even though it is correct and can pass any test cases. This is a good habit and helpful to improve our algorithm design and analysis skills.

5.Double Check Again
After optimize the algorithm, we should double check it again to ensure that it can pass all the possible test cases under the constraints of the problem. It is a good custom to be cautious all the time.

General Approach for Technical Problem Shooting

In our work (I mean computer science or relatives), we usually meet a lot of technical problems and some of them are difficult to deal with without any others’ help. In these cases, we have to do problem shooting all by ourselves. Here is the general approach that I used in my work or even daily life to solve technical problems. I think my approach can be also applicable in other area outside computer science.

  1. Master Basic Knowledge of the Area
    We should know the most common and basic knowledge of the area where the problems are encountered by us, otherwise we may even do not realize some issues have appeared. That may be dangerous in many areas, especially chemistry and nuclear. In computer science, the basic knowledge that we should master should include computer operating system, computer system architecture, computer programming, database, computer network, and programming language. This list may not be complete, but I think those aspects are essential in computer science. If we do not understand the solution provided by others, we should have to go back to pick up or start to learn the basic knowledge.

  2. Read or Watch the Tutorial (Document)
    When we use a new system or product in the first time, it is pretty easy for us to stuck in some problems. When it does happens, remember to read or watch the official instruction at first. Official instructions are the best tutorial in most of the time. Try to finger out the correct parts of the tutorial and use the solutions provided by it.

  3. Read or Watch Alternative Tutorial
    Sometimes we may find that official tutorial is not complete or even out of date! That is really bad for novices. In this case, we should find alternative tutorial through Internet. We can use Google search engine (www.google.com) or other powerful search engines to find the tutorial on the Internet. In most of the cases, we are not the first patch of users who use this product or system. Some one who has tried it before are probabily meet the same problems and find the solutions and post the solution on the Internet. So we can try to find these tutorials through Internet.

  4. Read or Watch Problem Shooting Approach
    In some cases, we may not find the tutorial on the Internet, but we can just find the solution dealing with problems. This is the first step when we meet a technical problem and this approach can save us lots of time because we can skip the lengthy and boring tutorials provided by others. But I do think reading or watching the tutorial or document should be first step. The reason is that tutorial can help us to know the every aspect of the product and thus, when next time we meet a different problem, we may be able to solve it without any help. But if we only know how to solve some specific problems, we may still stuck in other problems in the future. On average, reading or watching tutorials or document first do not cost more time and even save our time. In computer science, currently, stackoverflow and some forums are helpful for those to find approaches to solve their problems.

  5. Seek Help
    Unluckly, we may encouter some rare problems that no one meets before. In this case, we can post our problems on the Internet, send them in emails to experts or seek help from others through other approaches. In computer science, there are multiple forums and websites where people can seek help. They include stackoverflow, Quora and so on. Or we can suscribe our email address in the email list of the community manufactering the product, like Appache Hadoop, Hive, Mahout and so on. People in the community usually are willing to give us a hand and in most of the time reply the email in a short time. We can also seek help through social media like facebook and twitter or just make a phone call to those who may be able to solve the problems.

  6. Seek Help from People nearby
    If some one happens to be nearby, seek help from him/her, especially s/he is our colleague or classmate or friend or family member.

  7. Stop
    If we do fail to find a solution, we may have to give up and leave it alone if it does not harm.

Hope this blog will help :-)

Collections of Web Technology Concept

When developing Web application, it is very often to meet many new words when we read the document. Here is the collection of some most frequent concepts that Web developer will see in their work.

Cross-Domain Solution:
Here is the link of wikipedia:
http://en.wikipedia.org/wiki/Cross-domain_solution
A cross-domain solution (CDS) is a means of information assurance that provides the ability to manually or automatically access or transfer information between two or more differing security domains. They are integrated systems of hardware and software that enable transfer of information among incompatible security domains or levels of classification.
The three primary elements demanded from cross domain solutions are data confidentiality, data integrity and data availability.

Common Gateway Interface:
Here is the link of wikipedia:
http://en.wikipedia.org/wiki/Common_Gateway_Interface
A Common Gateway Interface(CGI) is a standard method used to generate dynamic content on Web pages and Web applications. It provides an interface between the Web server and programs that generate the Web content when implemented on a Web server. These programs are known as CGI scripts; they are usually written in a scripting language, or any programming language.

MIME:
Here is the link of its wikipedia:
http://en.wikipedia.org/wiki/MIME
MIME (Multi-Purpose Internet Mail Extensions) is an extension of the original Internet e-mail protocol that lets people use the protocol to exchange different kinds of data files on the Internet: audio, video, images, application programs, and other kinds, as well as the ASCII text handled in the original protocol, the Simple Mail Transfer Protocol (SMTP).
MIME as an Internet standard that extends the format of email to support: Text in character sets other than ASCII, Non-text attachments, Message bodies with multiple parts, and Header information in non-ASCII character sets.
Although MIME was designed mainly for SMTP, its use today has grown beyond describing the content of email and now often includes descriptions of content type in general, including for the web (see Internet media type) and as a storage for rich content in some commercial products (e.g., IBM Lotus Domino and IBM Lotus Quickr).

OAuth:
Here is the link of its wikipedia:
http://en.wikipedia.org/wiki/OAuth
OAuth is an open standard for authorization. OAuth provides client applications a ‘secure delegated access’ to server resources on behalf of a resource owner. It specifies a process for resource owners to authorize third-party access to their server resources without sharing their credentials.
This kind of authorization is often found in social media APIs.

REST:
Here is the link of its wikipedia:
http://en.wikipedia.org/wiki/Representational_state_transfer
Representational State Transfer (REST) is a software architecture style consisting of guidelines and best practices for creating scalable web services. REST is a coordinated set of constraints applied to the design of components in a distributed hypermedia system that can lead to a more performant and maintainable architecture.
REST has gained widespread acceptance across the Web[citation needed] as a simpler alternative to SOAP and WSDL-based Web services. RESTful systems typically, but not always, communicate over the Hypertext Transfer Protocol with the same HTTP verbs (GET, POST, PUT, DELETE, etc.) used by web browsers to retrieve web pages and send data to remote servers.
REST is a kind of architecture style instead of web standard.

SOAP:
Here is the link of its wikipedia:
http://en.wikipedia.org/wiki/SOAP
SOAP, originally an acronym for Simple Object Access protocol, is a protocol specification for exchanging structured information in the implementation of web services in computer networks. It uses XML Information Set for its message format, and relies on other application layer protocols, most notably Hypertext Transfer Protocol (HTTP) or Simple Mail Transfer Protocol (SMTP), for message negotiation and transmission.
SOAP has three major characteristics: extensibility, neutrality and independence.

Git Command Summary

Nowadays Github has become a widely used source version control and project management tool in software engineering. Thus, it is important to know the command lines of Github and use them in practice. Here is the summary of the most frequent used command lines of Github. If you find something inaccurate or missing, please contact me. :-)

Here is the link of Setting Up Git:
http://help.github.com/articles/set-up-git/

Here is the link of basic funcitons of Github, which include creating a repository, opening an issue, creating a branch, making a commit, opening a pull request and merge pull request:
https://guides.github.com/activities/hello-world/

Here are the frequent used Github commands and their explanation, referred from these links:
http://git-scm.com/docs
http://readwrite.com/2013/09/30/understanding-github-a-journey-for-beginners-part-1

git init

Initialize a new Git repository.

git clone

Clone a repository into a new directory, which is usually in local; and creates remote-tracking branches for each branch in the cloned repository, and creates and checks out an initial branch forked from the cloned repository’s current active branch.

git branch

List, create, or delete branches.

git config

Get and set repository and global options. Users can query/set/replace/unset options with this command.

git help

Display help information about Git. On default, a list of the most commonly used Git commands are printed out.

git status

Show the working tree status. Displays paths that have differences between the index file and the current HEAD commit, paths that have differences between the working tree and the index file, and paths in the working tree that are not tracked by Git.
The first one should be done by “git commit”; and last two ones should be done by “git add” and then “git commit”.

git add

Add files contents to the index found in the working tree, to prepare the content staged for the next commit.

git commit

Record changes to the repository. Stores the current contents of the index in a new commit along with a log message from the user describing the changes.
There are several ways to specify the content added:
(1) git add command
(2) git rm command
(3) listing files as arguments to the commit command so that their changes staged in the index will be ignored
(4) using -a switch with commit command to add changes from all known files and to remove files in the index removed from working tree
(5) using –iteractive or –patch switches to decide one by one which files or hunks should be part of commit.

git push

Update the remote refs along with associated objects, using local refs.

git pull

Fetch from and integrate with another repository or a local branch. Incorporates changes from a remote repository into the current branch.

git rm

Remove files from the working tree and from the index, but not from just working directory.

git mv

Move or rename a file, a directory, or a symlink.

git checkout //checkout是应用之意

Checkout a branch or paths to the working tree. Updates files in the working tree to match the version in the index or the specified tree.

git fetch

Download objects and refs from another repository. Fetch branches and/or tags from one or more other repositories, along with the objects necessary to complete their histories.

git merge

Join two or more development histories together. Incoporates changes from the named commits into the current branch.

git reset (–hard )

Rollback the respository to the given early commit.
