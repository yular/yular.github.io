Reading Note of Effective Java

Overall Summary

  • Clarity and simplicity are important and users of a module should never be surprised by its behavior.
  • Modules should be as small as possible but no smaller.
  • Code should be reused rather than copied.
  • The dependencies between modules should be kept to a minimum.
  • Errors should be detected as soon as possible after they are made, ideally at compile time.

Reading Note of The Pragmatic Programmer

I just finished reading a book called The Pragmatic Programmer, and found that this book was really helpful for entry level software engineer. I collected and then categorized the useful tips (personally I think) from that book, and noted and shared them here. And discussion is welcomed.

Tips with a question mark at the end means that I do not fully understand what the tip means.

Fundamental Guideline/Principle

  • DRY: Do not Repeat Yourself
    Need to get rid of following four kinds of duplication
    Imposed duplication Inadvertent duplication
    Impatient duplication Interdeveloper duplication
    Keep the the low-level knowledge in the code, where it belongs, and reserve the comments for other, high-level explanations.
    And the goal is :
    ** Make It Easy to Reuse

  • Orthogonality
    It means that signify a kind of independence or decoupling. Two or more things are orthogonal if changes in one do not affect any of the others.
    The goal is:
    Eliminate Effects Between Unrelated Things.
    To maintain orthogonality:     Keep code decoupled
    Avoid global data Avoid similar functions

Examples of orthogonality:
AOP CORBA

  • No Broken Windows
    The team should take the responsibility for the quality of product. High code quality can encourage developers to continue to write high quality of code for the project.

  • Boiled Frogs
    Make sure everyone actively monitors the environment for changes and solve the problems as soon as possible.

Understand the Goal/Requirement

  • There are NO final decisions.

  • Use Tracer Bullets to Find the Target
    Prototyping is the reconnaissance and intelligence gathering that takes place before a single trace bullet is fired.
    Tracer bullets should be a more concrete design plan for customers to view?

  • Prototype to Learn
    Here are things to prototype:
    Architecture New functionality in an existing system
    Structure or contents of external data Performance issues
    ** User interface design

  • Program Close to the Problem domain.

  • Do not Gather Requirement - Dig for them.

  • Work with a User to Think like a User.

  • Abstractions Live Longer than Details.

  • Use a Project Glossary.
    ** Project items dictionary.

Estimation

Estimates are based on models of given problem. Here are the basic steps to do estimation:

  • Understand what is being asked.

  • Build a model of the system.

  • Break the model into components.

  • Give each parameter a value.

  • Calculate the answers.

  • Keep track of estimating powers.

  • Estimating project schedules (Steps are as follows).
    Check requirements Analyze risk
    Design, implement, integrate Validate with the users

Tools

  • Keep knowledge in Plain Text

  • User the Power of Command Shells.

  • Use a Single Editor Well.
    It is better to know one editor very well, and use it for all editing tasks: code, documentation, memos, system administration, and so on.
    A good editor should have following features:
    Configurable Extensible
    Programmable Syntax highlighting
    Auto-completion Auto-indentation
    Initial code or document boilerplate IDE-like features

  • Always Use Source Code Control

  • Learn a Text Manipulation Language
    For example, awk and sed commands in Shell.

  • Write Code that Writes Code (Code Generator)
    Passive code generators are run once to produce a result. (Java Immutable class) Active code generators are used each time their results are required. (Any examples?)

Design

  • Design with Contracts.
    ** Shy contract - A way of separate concerns and decoupling.

  • Design by Contract.
    Focus on documenting (and agreeing to) the rights and responsibilities of software modules to ensure program correctness. Does no more and no less than the document claims to do.

  • The Law of Demeter for Functions
    ** Minimize Coupling Between Modules.

  • Analyze Workflow to Improve Concurrency

  • Design Using Service

  • Always Design for Concurrency

  • Use Blackboards to Coordinate Workflow
    ** Use the blackboard to coordinate disparate facts and agents, while still maintaining independence and even isolation among participants.

  • Some Things are Better Done than Described.

  • Do not be a Slave to Formal Methods.

  • Expensive Too Do Not Produce Better Design.

Coding/Implementation Practice

  • Crash Early.

  • If it Can not Happen, User Assertions to Ensure That It Will not.
    Precondition Check.

  • Use Exceptions for Exceptional Problems.

  • Finish What You Start
    The routine or object that allocates a resource should be responsible for deallocating it.

  • Dynamic Configuration: Configure, Do not Integrate.

  • Metadata-Driven Application: Put Abstractions in Code Details in Metadata.

  • Estimate the Order of Your Algorithms. And Test the estimates.

  • Do not program coincidence
    Need to know the idea behind the code clearly.

  • Do not be a slave to history. Do not let existing code dictate future code.
    Be critical to existing code.

  • Refactor (Following is the reason to do refactor)
    Duplication Nonorthogonal design
    Outdated knowledge Performance

  • Refactor early and often

  • Do not use Wizard Code You Do not Understand.

Best Practice(?)

  • Do not Use Manual Procedures.

Debug

  • Fix the Problem Not the Blame.

  • Do not panic.

  • Make bug reproducible.

  • Do not Assume It - Prove It.

  • Check my code first, and then bugs from third party.

Test

  • Test early, test often, test automatically.

  • Use Saboteurs to test your testing.
    It means that appoint a project saboteur, take a separate copy of the source tree, introduce bugs on purpose, and verify that the tests will catch them.

  • Test stage coverage, not code/line coverage

  • Find bugs once.
    Once a human tester finds a bug, it should be the last time a human tester finds that bug.

  • Coding is not done until all the tests run

  • Test against contract in unit test

  • Divide and conquer in testing

Documentation

  • Treat English as just another programming language.
    It means that it is important to keep the documentation in a good way.

  • Build documentation in. Do not bolt it on.(?)

Expectation

  • Gently Exceed Your Users’ Expectations.

  • Sign Your Work.
    Be pround of what your have done. (Ownership.)

Thinking in Number Theory

Number theory problems are very popular in algorithm programming contest. Most of time it is not easy to figure out the solution. Now here we will propose some generic ideas to think through this kind of problems.

This artical will be updated continously.

1. Prime Check

Given an integer, check whether it is a prime or not is a high frequent problem. There are a lot of algorithms to do this check. Some of them are as follows:

  • The naive O(n^1.5) algorithm:

  • Sieve of Eratosthenes.

  • Rabin-Miller primality test algorithm.

Sometimes, we need to figure out all the primes within a specific range. In that case, the second algorithm usually is the best choice, expecially for the problems described in the next sections.

2. Prime Factors

In Number Theory problems, we are ussually asked to figure out the factors of a given integer. In this case, a good idea (not always) is to figure out all its prime factors and their powers. To do that, we need to use (not always) Sieve of Eratosthenes to generate all the primes no larger than given integer. But this solution is not a silver bullet because sometimes the given integer is so large that we do not have enough memory to store all the primes or calculate them within the timebound.

3. Integer Generation

Maybe we find that it is inpractical to divide the given integer into mulplication of primes. Thus, we can try to multiply the primes we have to generate another integer. In this scenario, we do not have to worry about the limited memory to save large number of primes or find all the required primes in a short time. The reason is that even we multiply the same prime for several times (maybe around 30 or even less), we can get a very large integer. Remember, 2^64 = 18446744073709551616 .
