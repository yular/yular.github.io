Eight Golden Rules in Web UI Design

In Web user interface design, designers should follow some basic rules so that their work can be more attractive and have perfect user experience. Here are the most acknowledged eight golden rules for user interface design.

Strive for Consistency

This rule means:

  • Consistent sequence of actions for similar situations
  • Identical Terminology (including prompts, menus, help)
  • Consistent visual layout

Cater to universal usability

This rule means:

  • Recognize the needs of a diverse user group
  • Design for plasticity
  • Available from Novice to Expert, and covers the disabled.

Offer informative feedback

This rule means:

  • For every user action, the system should provide feedback
  • Frequency of task affects feedback type
  • Visual approaches make feedback easy

Design dialogs to yield closure

This rule means:

  • Action sequences should have a beginning, middle, and end.
  • Feedback provides sense of accomplishment

Prevent errors

This rule means:

  • Limit errors a user can make
  • In case of errors: detect error, using simple, constructive, and specific instructions, and do not change system state
  • Provide complete sequences, meaning that one action can perform a sequence of events, system should log errors and designers should study usage, error patterns, and user preferences via user groups, studies.
  • Universal usability can reduce errors

Permit easy reversal of actions

This rule means:

  • Actions should be reversible as much as possible
  • Design decision should include history size and what does it mean to undo something
  • Let the user know they can reverse an action

Support internal locus of control

This rule means:

  • Experiences operations want to feel in control
  • Builds anxiety and dissatisfaction because of surprising interface actions, tedious actions, difficulty in obtaining necessary ability and difficulty in producing action
  • Good rules: avoid acausality, make users initiators rather than responders

Reduce short-term memory load

This rule means:

  • Humans can remember 7 +/- 2 chunks of information
  • Display kept simple
  • Multiple page displays should be consolidated
  • Training if using codes, mnemonics, long sequence of actions
  • Online access to command-syntax, abbreviations, codes, etc.

E-commerce Business Model Summary

Here is the summary of current most widely used business models in E-Commerce. They include B2B, B2C, C2C, P2P and O2O. Their definitions, difference and sample companies are as follows.

1.B2B
In a Business-to-Business E-commerce environment, companies sell their online goods to other companies without being engaged in sales to consumers.

2.B2C
In a Business-to-Consumer E-commerce environment, companies sell their online goods to consumers who are the end users of their products or services. Usually, B2C E-commerce web shops have an open access for any visitor, meaning that there is no need for a person to login in order to make any product related enquiry.

Example: Amazon

3.C2C
In a Consumer-to-Consumer E-commerce environment consumers sell their online goods to other consumers.

Example: eBay

4.C2B
In a Consumer-to-Business E-commerce environment, consumers usually post their products or services online on which companies can post their bids. A consumer reviews the bids and selects the company that meets his price expectations.

4.P2P(L/I â¦)
Peer-to-peer lending commonly abbreviated as P2PL is the practice of lending money to unrelated individuals, or âpeersâ, without going through a traditional financial intermediary such as a bank or other traditional financial institution. P2PL is not to be confused with peer to peer investing (P2PI). This lending takes place online on peer-to-peer lending companies websites using various different lending platforms and credit checking tools.

Example: Lending Club

5.O2O
Online To Offline. A kind of offline business model. Here is the definition in Wiki, but only in Chinese.

Example: Uber and Airbnb

Object-Oriented Design Pattern Summary

This summary is the note of Head First Design Pattern, including some basic knowledge, guidelines, and all the design patterns and their application examples described in the book.

Here is the link of Head First Design Pattern in Amazon:
Head First Design Pattern

###Basic

#####1. Basic Knowledge
[1]Design Pattern Concept
A Pattern is a solution to a problem in a context.

[2]Pattern Categorites

  • Creational patterns
  • Structural patterns
  • Behavioral patterns

[3]Object Oriented Basic

  • Abstraction
  • Encapsulation
  • Polymorphism
  • Inheritance.

[4]Object Oriented Principles

  • Encapsulate what varies; Favor composition over inheritance; Program to interfaces, not implementation.
  • Strive for loosely coupled designs betweens objects that interact. (The two objects have very little knowledge of each other)
  • Classes should be open for extension but closed for modification.
  • Dependency Inversion Principle: Depend upon abstraction. Do not depend on concrete classes.
  • Least Knowledge: Talk only to your immediate friends.
    Note: Only invoke methods that belongs to: the object itself; objects passed in as a parameter to the method; Any object the method creates or instantiates; Any components of the object.
  • Hollywood Principle: Do not call us, we will call you.
  • A class should have only one reason to change.

[5]Tips:

  • Design Patterns give us a shared vocabulary with other developers and let us think at the pattern level.
  • A design guru thinks about how to create flexible designs that are maintainable and that can cope with change.

###GuideLines

#####2. Guidelines to Follow the Dependency Inversion Principle

  • No variable should hold a reference to a concrete class.(Use a factory to get around that)
  • No class should derive from a concrete class
  • No method should override an implemented method of any of its base classes.

#####3. Pattern Design Writers Guidelines
[1]To be a Design Pattern Writer

  • Do your homework
  • Take time to reflect and evaluate
  • Get your ideas down on paper in a way others can understand
  • Have others try your patterns then refine and refine some more
  • Do not forget the rule of three

[2]Better Living with Design Pattern

  • Learn the all too common misconceptions about the definition of a âDesign Patternâ
  • Discover those unify Design Pattern Catalogs and why you just have to get one
  • Avoid the embarrasment of using a Design Pattern at the wrong time
  • Learn how to keep patterns in classifications where they belong
  • See that discovering patterns is not just for the guius: read out quick How To and become a patterns writer too
  • Be there when the true identity of the mysterious Gang of Four is revealed
  • Keep up with the neighbours - the coffee table books any patterns user must down
  • Learn to train your mind a Zen master
  • Win friends and influence developers by improving your patterns vocabulary

[3]Top five ways to share your vocabulary

  • In design meetings
  • With other developers
  • In architecture documentation
  • In code comments and naming conventions
  • To groups of interested developers

[4]Thinking in Patterns

  • Keep it Simple
  • Design Patterns are not a magic bullet; In fact, they are not even a bullet
  • You know you need a pattern when you find that a simple solution does not work.
  • Refactoring time is Patterns time!
  • Take out what you do not really need. Do not be afraid to remove a Design Pattern from the design
  • If you do not need it now, do not do it now

[5]Mind on Patterns

  • Beginner Mind: The beginner uses patterns everywhere.
  • Intermediate Mind: As learning progresses, the intermediate mind starts to see where patterns are needed and where they are not.
  • Zen Mind: Be able to see where patterns fit naturally.

###Design Patterns

#####4. The Strategy Pattern
The Strategy Pattern defines a family of algorithms encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from clients that use it.

Example: Duck Behaviors Design

Tips:
[1]Identify the aspects of application that vary and separate them from what stays the same.

  • âProgram to an interfaceâ really means âProgram to a supertypeâ. (Polymorphism)
    When abstract superclass or when interface?
  • Relationship between classes: HSA-A, IS-A and IMPLEMENTS
  • The promise of the object-oriented design(way) is reuse.

#####5. The Observer Pattern
Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

Note: Loosely coupled is needed.

#####6. The Decorator Pattern
Attach additional responsibility to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

Example: Java IO classes

Note: Nesting construction

#####7. The Factory Pattern
Defines an interface for creating an object, but less subclasses decide which class to instantiate. Factory Method lets a class defer instantiation to subclasses.

Note:

  • Creator Class and Product Class, Parallel Class Hierarchies;
  • Encapsulating object creation;
  • Depend on abstraction instead of concrete classes;
  • Subclass

#####8. The Abstract Factory Pattern
Provides an interface for creating families of related or dependent objects without specifying their concrete classes.

Note:

  • Often implemented as factory methods;
  • Used when have families of products needed to create

#####9. The Singleton Pattern
Ensures a class has only one instance and provides a global point of access to it.

#####10. The Command Pattern
Encapulation a request as an object, thereby letting you parameterize clients with different requests, queue or log requests and support undoable operation.

Note:
Support Queue data structure.

#####11. The Adapter Pattern
Converts the interface of a class into another interface the clients expect. Adapter lets classes work together that could not otherwise because of incompatible intefaces.

Example: Home Theater

#####12. The Facade Pattern
Provides a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.

Note:
To make a system easier to use through a simplied interface.

#####13. The Template Method Pattern
Defines the skeleton of an algorithm in a method, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps of an algorithm without changing the structure of algorithm.

Example: JFrames swinging

#####14. The Iterator Pattern
Provides a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

Example: Java.util Iterator, Java Collection Framework.

#####15. The Composite Pattern
Allows you to compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

#####16. The State Pattern
Allows an object to alter its behavior when its internal state changes. The object will appear to change its class.

#####17. The Proxy Pattern
Provides a surrogate or placeholder for another object to control access to it.

Example: Java.rmi; Remote Proxy && Virtual Proxy && Protect; Proxy-Java Reflection.

#####18. The Compound Pattern (Pattern of Pattern)
A Compound Pattern combines two or more patterns into a solution that solves a recurring or general problem.

Example: Model-View-Control.

#####19. The Anti-Pattern
How to go from a problem to a BAD solution

Extra: Some Simple Steps of Designing Gumball Machine
[1]Define a State interface that contains a method for every action in the Gumball Machine.
[2]Then we are going to implement a State class for every state of the machine. These classes will be responsible for the behavior of the
machine when it is in the corresponding state.
[3]Finally, we are going to get rid of all of our conditional code and instead delegate to the state class to do the work for us.
