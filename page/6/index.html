Thinking in Number Theory

Number theory problems are very popular in algorithm programming contest. Most of time it is not easy to figure out the solution. Now here we will propose some generic ideas to think through this kind of problems.

This artical will be updated continously.

1. Prime Check

Given an integer, check whether it is a prime or not is a high frequent problem. There are a lot of algorithms to do this check. Some of them are as follows:

  • The naive O(n^1.5) algorithm:

  • Sieve of Eratosthenes.

  • Rabin-Miller primality test algorithm.

Sometimes, we need to figure out all the primes within a specific range. In that case, the second algorithm usually is the best choice, expecially for the problems described in the next sections.

2. Prime Factors

In Number Theory problems, we are ussually asked to figure out the factors of a given integer. In this case, a good idea (not always) is to figure out all its prime factors and their powers. To do that, we need to use (not always) Sieve of Eratosthenes to generate all the primes no larger than given integer. But this solution is not a silver bullet because sometimes the given integer is so large that we do not have enough memory to store all the primes or calculate them within the timebound.

3. Integer Generation

Maybe we find that it is inpractical to divide the given integer into mulplication of primes. Thus, we can try to multiply the primes we have to generate another integer. In this scenario, we do not have to worry about the limited memory to save large number of primes or find all the required primes in a short time. The reason is that even we multiply the same prime for several times (maybe around 30 or even less), we can get a very large integer. Remember, 2^64 = 18446744073709551616 .

Thinking in Test

This summary may be updated in the future.

We need to follow some steps to test service starting from development to deployment. It is very important because it can help us to find issues and fix them as soon as possible before customer experience become worse.

Roughly, there are maily Four steps in the through test on service. They are tests/verification in (local) development, tests while building service in remote machines, tests in different stages of deployment on remote machines, and manual verification of services on remote machines.

Here are some details of these steps.

Local Tests

When develop service on the local, we need to test it before build and deploy it on remote machines. In local development, at least we need these tests:

  • Unit Tests
  • Integration Tests (Test integration between service and data storage or among different components of the service)
  • Manual Verification

In this step, we need to check test coverage including class coverage, method coverage and line coverage.

Tests While Building Software on Remote Machines

Before we actually deploy service on remote machines, we need to build it. While building service, we need to run the tests which are written in development. These tests are unit tests and integration tests in the last stage.

In this stage, the building process stops if any tests fail.

Tests After Deploying Software on Remote Machines

After deploy service on the remote machines successfully, we need to test the public APIs or performance of the service in all the stages of deployment. Different stages may include Beta stage, Gamma stage, PreProd stage and Prod stage or even more stages.

In these stages, we need to run following tests: Load Tests, Fuzz Tests, Code Review Verification and tests gathering performance statistics.

Manual Verification After Deployment of Remote Machines

Even though we deploy service on remote machines successfully, we still need to manually test the service. One way is to test APIs through Web GUI or Command Line Interface. Besides, we can manually change the data storage or the behaviors of system to simulate real world situation.

The basic process is that we need to create a .plist (plist scripts) text file, add timed jobs setting inside the file, put that file under certain directory, and then use launchctl command to start timed jobs.

Choose Timed Jobs Directory

Mac will startup timed jobs that defined in .plist files under certain directories only. Here is the list of those directories:

  • /Library/LaunchDaemons - Put our plist scripts in this folder if our job needs to run even when no users are logged in.
  • /Library/LaunchAgents - Put our plist scripts in this folder if the job is only useful when users are logged in.
  • $HOME/Library/LaunchAgents - Put our plist files in this folder if the job is only useful when users are logged in.

Note that we need to use sudo command if we want to put plist scripts under first two directories.
