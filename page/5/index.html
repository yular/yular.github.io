Reading Note of Effective Java I

Item 1. Consider static factory methods instead of constructors

Static factory method, which is different from regular factory pattern. With a well-chosen name is easier to use and the resulting client code easier to read. Not required to create a new object each time they are invoked, singleton factory for example. Can return an object of any subtype of their return type. Requires the client to refer to the returned object by its interface rather than its implementation class, which is generally good practice. Reduce the verbosity of creating parameterized type instances.

But this pattern has some disadvantages. Main disadvantage is that classes without public or protected constructors cannot be subclassed. They are not readily distinguishable from other static methods.

Item 2. Consider a builder when faced with many constructor parameters.

Builder is a static class.

The main disadvantage of a builder is that cost of creating builder is unlikely to be noticeable.

Item 3. Enforce the singleton property with a private constructor or an enum type.

Making a class a singleton can make it difficult to test its clients as it is impossible for mocking unless an interface is provided.

Item 4. Enforce noninstantiability with a private constructor.

This kind of classes are utility class, which group static methods and static fields. Need to add comments on these classes.

Item 5. Avoid creating unnecessary objects.

Reuse a single object if possible. A reusable object should be immutable. And static factory methods are better choice to create reusable objects.

Item 6. Eliminate obsolete object references (Otherwise memory leak may happen)

Two ways to do:

  • Exception rather than norm: Null out references once they become obsolete.
  • Best way: Let the variable containing the reference fall out of scope.

Source of memory leak:

  • A class manages its own memory.
  • Cache. Solution: use weak hashmap.
  • Listeners and other callbacks: store only weak references to them, like keys in weak hashmap.

Item 7. Avoid finalizers

Finalizers are unpredictable, often dangerous and generally unnecessary. Avoid using finalizer() method in Java:

  • Finalizer may not be executed promptly and finalizer queue can be low priority in some JVM environment.
  • Never depends on a finalizer to update critical persistent state because program can terminate without executing finalizers.
  • There is a severe performance penalty for using finalizer. Never do time critical tasks in finalizer().

Instead, provide an explicit termination method. Explicit termination methods are typically used in combination with the try-finally construt to ensure termination.

Finalizer can be used to log a warning if it finds that the resource has not been terminated.

Finalizer can be used as a safety net in case of explicit termination methods are not called.

Use finalizer to terminate noncritical native resources.

Remember to invoke super.finalize when have to use finalizer.

Item 8. Obey the general contract when overriding equals

  • Each instance of the class is inherently unique.
  • Do not care whether the class provides a logical equality test.
  • A super class has already overriden equals and its behavior is appropriate for this class.
  • The class is private or package-private, and you are certain that its equals methods will never be invoked.

equals method general contract:

  • Reflexive: x.equals(x) must be true.
  • Symmetric: x.equals(y) is true then y.equals(x) must be true.
  • Transitive: x.equals(y) and y.equals(z) are true then x.equals(z) is true.
  • Consitent: x.equals(y) is the same result always unless object is modified.

Liskov substitution principle says that any important property of a type should also hold for its subtypes so that any methods written for the type should work equally well on its subtypes.

Important notes to write equals method:

  • Can add a value component to a subclass of an abstract class without violating the equals contract.
  • Do not write an equals method depending on unreliable resources.
  • Non-nullity: all objects must be unequal to null.
  • A high quality equals method: use == to check if the argument is a reference to this object. Use the instanceof operator to check if the argument has the correct type; Cast the argument to the correct type. For each significant field in the class, check if that field of the argument matches the corresponding field of this object.
  • When finished writing equals method, check if it is symmetric, transitive and consistent.

Item 9. Always override hashCode when you override equals

  • Do not try to be too clever.
  • Do not substitute another type for Object in the equals declaration.
  • Equal objects must have equal hashcodes.
  • Do not be tempted to exclude significant parts of an object from the hashcode computation to improve performance.

Item 10. Always override toString

  • It is recommended to override toString and even in subclasses.
  • In practice, toString method should return all of the interesting information contained in the object.
  • Whether or not decide to specify the format of toString return value, clearly document intentions, provide programmatic access to all of the information contained in the value returned by toString.

Reading Note of Java in a Nutshell (Java 7 and 8)

Here is the summary of the book Java in a Nutshell (JDK 7 and 8 version).

JVM and Java Platform

JVM executes Java program in Java bytecode format, which is called class files - the ones with .class extension. Java bytecode is an intermediate code which is between source code and machine code.

Java platform uses just-in-time compilation to take the advantage of the fact that parts of Java program are called more frequently.

Two ways to run other languages on JVM:

  • Have a source code compiler producing class files.
  • Implement an interpreter and runtime in Java, and then interpret the source form of the language.

Software Construction

There are mainly elevent steps as below:

  • Problem Definition
  • Requirement Development
  • Construction Planning
  • Software Architecture or High Level Design
  • Detailed Design
  • Coding and Debugging
  • Unit Testing
  • Integration Testing
  • Integration
  • System Testing
  • Corrective Maintenance

Coding and Style

Singleton

  • Use a framework (usually dependency injection) to provide singleton automatically.

Lambda expressions:

  • A lambda expression is an anonymous collection of executable Java code, essentially a method body.
  • Syntax of lambda expressions is like: ( paramlist ) -> { statements }.
  • When a lambda expression is used as a value, it is automatically converted to a new object of the correct type for the variable being placed into. This autoconversion and type inference is essential to Java’s approach to lambda expression.

Naming Convention:

  • Class: Class names are nouns to represent objects.
  • Interface: Interface names are adjective if to provide additional information about the class, otherwise are nouns to work like abstract superclasses.

Label statements:
A label statement is simply a statement that has been given a name by prepending an identifer and a colon to it.

Generic type:
For wildcard, it is legal to write in following style: List<?> objects = new ArrayList(). List is subtype of List<?>. But List<?> is not a subtype of any List.

Reflection (Convert specific to general):

  • Only deal with cases immediately applicable to problem domain.
  • To use reflection: Class<?> clz = getSomeClassObject(); Object rcvz = clz.newInstance();

Datatype Overflow:
Integer arithmetic never overflow or underflow because numbers just wrap around.

Handling Exception

Exceptions are Throwable objects and that exceptions fall into two main categories, specified by the Error and Exception subclass. Any exception object that is an Error is unchecked.

Security

A safe program includes:

  • All objects start off in a legal state after creation.
  • Externally accessible methods transition objects between legal state.
  • External accessible methods must not return with object in an inconsistent state.
  • External accessible methods must reset object to a legal state before throwing (exceptions).

Use java.util.Arrays.equals to compare the content of two arrays.

Concurrency

Two collectors:

  • Parallel collectors, using multiple threads to perform collections.
  • Concurrent collectors, running at the same time as application threads are still running.

IO

  • Use Readers and Writers to do IO operation.
  • Use try-with-resources (TWR) in IO operation to take care of resources management. TWR uses a new interface called AutoCloseable.
    TWR style is like: try(InputStream is = new FileInputStream(“”)) { … }
  • Use File class for file operation and Path class for file system path operation.
  • For high-performance IO, use ByteBuffer.
  • For Async IO, use AsynchronousFileChannel for file IO, AsynchronousSocketChannel for client socket IO, and AsynchronousServerSocketChannel for asynchronous sockets accepting incoming connection.
  • For network, use java.net or javax.net packages.
  • For Async style, use Future-Based style, Callback-Based style and watchservice on a directory.
  • Use MappedByteBuffer to store a memory-mapped file into memory.

Note :

  • ByteBuffer and MappedByteBuffer are NIO buffers which are low-level abstraction for high-performance IO.

Time

Use Java 8 Date and Time API: java.time .

Garbage Collection

Concurrent Mark and Sweep (CMS):
Only used to collect the old generation-conjunction with a parallel collector that is to clean young generation.

G1:
Used in high throughput environment and allows users to specify pause goals, which is low-pause collector.

Command Line Interface

Here are the most frequently used Java commands:

  • javac
  • java
  • jar
  • javadoc
  • jdeps
  • jps
  • jstat
  • jstatd
  • jinfo
  • jstack
  • jmap
  • javap

Miscellaneous

  • Nashorn
  • JVisualVM

Linux Text Processing Command I

sort

  • File sort utility, often used as a filter in a pipe.

tsort

  • Topological sort.

uniq

  • Remove duplicate lines from a sorted file.

expand, unexpand

  • expand: a filter converting tabs to spaces. Ofen used in a pipe.
  • unexpand: reverse effect of expand.

cut

  • Widely used for text processing.
  • A tool for extracting fields from files. Can be used in a pipe.

paste

  • Tool for merging together different files into a single, multi-column file.
  • When combining with cut, useful for creating system log files.

join

  • Tool for merging two files in a meaningful fashion, creating a simple version of relational database.

head

  • Lists the begining of a file to stdout.

tail

  • List the end of a file to stdout.
  • With -f option, useful for monitoring system.

grep

  • A multi-purpose file search tool that uses Regular Expressions.
  • Command pattern: grep pattern [files …]
  • Very useful in pipe.
  • -i: a case sensitive search.
  • -w: matches only whole words.
  • -r: searches files recursively.
  • -n: lists the matching lines.
  • -v: filters out matches.
  • -c: gives a numerical count of matches.
  • –color: marks the matching string in color.

agrep

  • Approximate grep, extending the capabilities of grep to approximate matching.

look

  • Works like grep, but does a look up on a “dictionary”, a sorted word list in /usr/dict/words by default.

sed

  • Non-interactive stream editor, permitting using many ex commands in batch mode.
  • Useful in shell scripts and pipe.

awk

  • Programmable file extractor and formatter, good for manipulating and/or extracting fields(columns) in structured text files. Has syntax similar to C.

wc

  • Give a “word count” on a file or I/O stream.

tr

  • Character translation filter.
  • Must use quoting and/or brackets as appropriate.

fold

  • A filter that wraps lines of input to a specified width.

fmt

  • Simple-minded file formatter, used as a filter in a pipe to wrap long lines of text output.

col

column

  • Column formatter.

colrm

  • Column removal filter.

nl

  • Line numbering filter.

pr

  • Print formatting filter.

gettext

  • A set of utilities for localizing and translating the text output of programs into foreign languages.

msgfmt

iconv

recode

TeX,gs

texexec

enscript

groff, tbl, eqn

lex, yacc

  • Lexical analyzer commands.

Java Non Access Modifiers Summary

Here is a short summary of Java non access modifiers, which include static, final, abstract, synchronized, transient and volatile. Here are the references: JSL8 and Java Non Access Modifiers from TutorialsPoint.

Static Modifier

Static modifier can be used as variable and method modifier.

Static Variable

Static variables will exist independently for all the instances of the class. There is only one copy of every static variable in the memory. It is a kind of class variable and local variables cannot be declared static.

Static Method

Static methods will exist independently for all the instances of the class. Static methods can be used directly without any instance variables of any object of the class they are defined in.

Final Modifier

Final modifier is used to finalize the implementation/creation of variales, methods, and classes.

Final Variable

Every final variable can be explicitly initialized only once and then they cannot be reassigned to refer to a different object. But even though final variables use the same reference all the time, the value of their members can be changed.

Usually, final and static modifiers are used together to define constants.

Final Method

Final methods cannot be overriddedn by any subclasses. It indicates that the content of these methods should not be changed by any clients.

Final Class

Final classes cannot be extended by other classes. It indicates that these classes should not be inherited.

Abstract Modifier

Abstract modifier can be used for methods and classes, indicating "change" the content of these methods or classes in subclasses.

Abstract Method

Abstract methods do not have method body and their implementation should be provided by subclasses. Abstract methods can never be final or strict.
If a class has one or more abstract methods, it must be declared as abstract.

Abstract Class

Abstract classes cannot be instantiated and they are designed for inheritance. Abstract classes can never be final and they can contain both abstract methods and normal methods.

Synchronized Modifier

Synchronized modifier can be used for methods and applied on variables (lock a specific variable). It indicates that these methods can be accessed by only one thread at a time. Synchronized modifier is widely used in concurrent situation.

Transient Modifier

Transient modifier can be used for variables. It indicates that JVM will skip these variables when it is serializing the objects containing them.

Volatile Modifier

Volatile modifier can be used for variables. It indicates that a thread accessing these variables must always merge its own copy of them with the master copy in the memory.
Volatile modifier can only be used for variables in instances, which are of type object or private. A volatile object reference can be null.

Reading Note of Test Driven Development by Example

Here is the summary of Test Driven Development book.

Overall Rules

  • In Test-Driven Development, there are two basic rules:

    • Write new code only if there is a failing automated test.
    • Eliminate duplication.

  • The two rules above imply the order of tasks of programming:

    • Red: Write a little test that does not work, even does not compile at first
    • Green: Make the test work quickly, committing whatever sins necessary in the process
    • Refactor: Eliminate all the duplication created in just getting the test to work

  • The three basic steps of Test Driven Development are:

    • Start simply
    • Write automated tests
    • Refactor to add design decisions one at a time

  • Two topics where TDD is not good at:

    • Security software
    • Concurrency

  • Rhythm of TDD:

    • Quickly add a test
    • Run all tests and see the new one fails
    • Make a little change
    • Run all tests and see them all succeed
    • Refactor to remove duplication

  • Three ways to pass tests:

    • Fake it.
    • Obvious implementation.
    • Translate a design object (side effects) into a test case that fails because of the objection, get the code to compile quickly with a stub implementation, made the test work by typing in what seemed like the right code.

  • Given a problem, first divide it to small independent problems, create tests for each of them, then do refactor on the code.

Detailed Rules

  • Write new code only if you first have a failing automated test. Then eliminate the duplication.

  • Could not tackle a big test, so invent a small test representing progress, write the test by shameless duplicating and editing, make the test work by copying and editing model code wholescale, promised ourselves to remove duplication.
