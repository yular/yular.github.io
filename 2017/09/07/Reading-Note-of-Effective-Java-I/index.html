Item 1. Consider static factory methods instead of constructors

Static factory method, which is different from regular factory pattern. With a well-chosen name is easier to use and the resulting client code easier to read. Not required to create a new object each time they are invoked, singleton factory for example. Can return an object of any subtype of their return type. Requires the client to refer to the returned object by its interface rather than its implementation class, which is generally good practice. Reduce the verbosity of creating parameterized type instances.

But this pattern has some disadvantages. Main disadvantage is that classes without public or protected constructors cannot be subclassed. They are not readily distinguishable from other static methods.

Item 2. Consider a builder when faced with many constructor parameters.

Builder is a static class.

The main disadvantage of a builder is that cost of creating builder is unlikely to be noticeable.

Item 3. Enforce the singleton property with a private constructor or an enum type.

Making a class a singleton can make it difficult to test its clients as it is impossible for mocking unless an interface is provided.

Item 4. Enforce noninstantiability with a private constructor.

This kind of classes are utility class, which group static methods and static fields. Need to add comments on these classes.

Item 6. Eliminate obsolete object references (Otherwise memory leak may happen)

Two ways to do:

  • Exception rather than norm: Null out references once they become obsolete.
  • Best way: Let the variable containing the reference fall out of scope.

Source of memory leak:

  • A class manages its own memory.
  • Cache. Solution: use weak hashmap.
  • Listeners and other callbacks: store only weak references to them, like keys in weak hashmap.

Item 7. Avoid finalizers

Finalizers are unpredictable, often dangerous and generally unnecessary. Avoid using finalizer() method in Java:

  • Finalizer may not be executed promptly and finalizer queue can be low priority in some JVM environment.
  • Never depends on a finalizer to update critical persistent state because program can terminate without executing finalizers.
  • There is a severe performance penalty for using finalizer. Never do time critical tasks in finalizer().

Instead, provide an explicit termination method. Explicit termination methods are typically used in combination with the try-finally construt to ensure termination.

Finalizer can be used to log a warning if it finds that the resource has not been terminated.

Finalizer can be used as a safety net in case of explicit termination methods are not called.

Use finalizer to terminate noncritical native resources.

Remember to invoke super.finalize when have to use finalizer.

Item 8. Obey the general contract when overriding equals

  • Each instance of the class is inherently unique.
  • Do not care whether the class provides a logical equality test.
  • A super class has already overriden equals and its behavior is appropriate for this class.
  • The class is private or package-private, and you are certain that its equals methods will never be invoked.

equals method general contract:

  • Reflexive: x.equals(x) must be true.
  • Symmetric: x.equals(y) is true then y.equals(x) must be true.
  • Transitive: x.equals(y) and y.equals(z) are true then x.equals(z) is true.
  • Consitent: x.equals(y) is the same result always unless object is modified.

Liskov substitution principle says that any important property of a type should also hold for its subtypes so that any methods written for the type should work equally well on its subtypes.

Important notes to write equals method:

  • Can add a value component to a subclass of an abstract class without violating the equals contract.
  • Do not write an equals method depending on unreliable resources.
  • Non-nullity: all objects must be unequal to null.
  • A high quality equals method: use == to check if the argument is a reference to this object. Use the instanceof operator to check if the argument has the correct type; Cast the argument to the correct type. For each significant field in the class, check if that field of the argument matches the corresponding field of this object.
  • When finished writing equals method, check if it is symmetric, transitive and consistent.

Item 9. Always override hashCode when you override equals

  • Do not try to be too clever.
  • Do not substitute another type for Object in the equals declaration.
  • Equal objects must have equal hashcodes.
  • Do not be tempted to exclude significant parts of an object from the hashcode computation to improve performance.

Item 10. Always override toString

  • It is recommended to override toString and even in subclasses.
  • In practice, toString method should return all of the interesting information contained in the object.
  • Whether or not decide to specify the format of toString return value, clearly document intentions, provide programmatic access to all of the information contained in the value returned by toString.
