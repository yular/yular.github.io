Thinking in Github Usage

Github is very popular in software industry nowadays. The following is a short summary of how to effectively use Github for source code control.

Branch Mangement Strategy

When we create a repository in Github, we will have a master branch for it by default. Usually we should have these four kinds branches:

  1. Master branch: This branch should manage the stable version of the project.
  2. Development branch: This branch should manage the latest version but no necessary the stable version of the project. Every developers should commit their changes to this branch and then these changes will be merged to master branch after this latest version pass the test.
  3. Personal branch: It is good practice that every developer should have their own development branch. They develop new feature or fix bugs in this branch and then merge the changes to the development branch after their code pass test and code review.
  4. Bug branch: This branch is temporary. When developers are asked to fix an emergent bug(s) but at the same time they are developing other features, they can create a bug branch to fix this bug, commit and merge the changes, and at last, delete this branch.
  5. Feature branch: This branch is also tempory. Here is a similar story: Developers again are asked to add a new feature that should be released in a short time, but at the same time they are working on different features or even modules. This should be a daily event in IT company :-) So again, we can create a branch only for the purpose of developing this feature, commit and merge the changes and then delete this branch.
    Actually, there should be more different kinds of branches and we should create, manage and delete the branches based on project management need.
